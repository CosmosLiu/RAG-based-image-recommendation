# Image RAG 项目代码分析文档

## 项目概览

这是一个基于 **RAG（检索增强生成）** 思想的**图片智能搜索与推荐系统**。核心流程为：利用多模态大模型（Ollama + Qwen-VL）对图片进行多维度语义理解，将文本特征通过 Embedding 模型转换为高维向量存入 Milvus 向量数据库，并支持通过文本描述或图片进行混合检索召回。

**技术栈：** FastAPI + Milvus + Ollama（qwen3-vl:8b / bge-m3）

---

## 模块划分

项目共包含 **4 个 Python 模块** 和 **1 个依赖声明文件**：

| 文件 | 职责 |
|------|------|
| `schemas.py` | API 请求/响应的数据模型定义 |
| `feature_service.py` | 调用 Ollama 进行图片分析、文本扩写和向量化 |
| `milvus_db.py` | Milvus 向量数据库的增、查、混合检索操作 |
| `main.py` | FastAPI 应用入口，定义 3 个 REST API 端点 |
| `requirements.txt` | 项目依赖声明 |

---

## 模块详细分析

### 1. `schemas.py` —— 数据模型层

**职责：** 定义 API 接口的请求体与响应体结构，使用 Pydantic 进行数据校验。

**包含 4 个模型：**

| 类名 | 类型 | 字段 | 说明 |
|------|------|------|------|
| `SaveImageRequest` | 请求体 | `uid: str`，`image_url: str` | 图片入库请求 |
| `TextRecommendRequest` | 请求体 | `uid: str`，`query: str`，`top_k: int = 10` | 文本搜图请求，默认返回 10 张 |
| `ImageRecommendRequest` | 请求体 | `uid: str`，`image_url: str`，`top_k: int = 10` | 以图搜图请求，默认返回 10 张 |
| `RecommendResponse` | 响应体 | `results: List[str]` | 推荐结果，返回图片 URL 列表 |

**设计特点：**
- 所有请求都绑定 `uid` 字段，实现**用户级数据隔离**，不同用户的图片库互不干扰。
- `top_k` 有默认值 10，调用方无需必传。

---

### 2. `feature_service.py` —— AI 特征提取服务层

**职责：** 封装所有与 Ollama 大模型的交互逻辑，包括图片分析、文本意图扩写、文本向量化三个核心能力。这是整个系统的"智能大脑"。

#### 配置区

- `BASE_URL = "http://localhost:11434/v1"` — Ollama 的 OpenAI 兼容接口
- `VISION_MODEL = "qwen3-vl:8b"` — 视觉理解模型，处理图片分析及文本扩写
- `EMBEDDING_MODEL = "bge-m3"` — 文本向量化模型，输出 1024 维向量

#### 核心函数

##### `get_embedding(text, name) -> list[float]`

调用 Ollama 的 `/v1/embeddings` 接口，将任意文本转换为 1024 维浮点数向量。向量是后续相似度检索的基础。打印耗时便于性能观测。

##### `call_chat_completion(messages, model) -> dict`

通用的 LLM 对话调用封装，向 Ollama 发送 messages 并要求以 JSON 格式返回。内置容错：如果模型返回的不是合法 JSON，会降级处理，把原始内容塞入 `description` 字段。

##### `get_image_base64(image_url) -> str`

从 HTTP/HTTPS URL 下载图片并转换为 Base64 编码的 Data URI，供视觉模型识别。这是 Ollama 多模态接口的标准输入格式。

##### `extract_features_from_image(image_url) -> dict`

**图片入库的核心流程（步骤 1-2）：**

1. 下载图片 → 转 Base64
2. 调用 qwen3-vl 视觉模型分析图片，从三个维度提取语义：
   - **visible_context（显性特征）：** 图片中的客观元素（物体、人物、颜色、形状、文字等）
   - **hidden_context（隐性特征）：** 图片传达的氛围、情感、艺术风格、深层含义
   - **description（整体描述）：** 对整张图片的详细综合描述
3. 将三路文本分别通过 bge-m3 模型转为 1024 维向量

**返回值：** 6 个字段 —— 三路原始文本 + 三路向量

##### `extract_features_from_text(query) -> dict`

**文本搜图的核心流程（步骤 1-2）：**

1. 接收用户的简短搜索词
2. 调用 LLM（文本模式）将搜索词"扩写"成三个维度的描述：
   - 预期可见的显性特征关键词
   - 预期传达的隐性氛围/风格关键词
   - 预期匹配的理想图片完整描述
3. 将三路文本分别转为向量

**设计亮点：** 用户输入的简短搜索词（如"温暖的午后"）会被 LLM 扩写为丰富的多维特征，大大提高了向量检索的召回精度。

### 3. `milvus_db.py` —— 向量数据库层

**职责：** 封装 Milvus 向量数据库的所有操作，包括集合管理、数据插入和**混合检索（Hybrid Search）**。

#### 类 `MilvusModel`

##### `__init__(uri, db_name, collection_name, auto_create, load_collection)`

- 连接 Milvus 服务（默认 `localhost:19530`）
- 若集合不存在且 `auto_create=True`，自动创建集合
- 若 `load_collection=True`，将集合加载到内存

##### `_create_schema()` —— 定义集合的字段结构

| 字段名 | 类型 | 长度/维度 | 说明 |
|--------|------|-----------|------|
| `id` | INT64 | — | 主键，自增 |
| `uid` | VARCHAR | 64 | 用户唯一标识 |
| `image_id` | VARCHAR | 512 | 图片 URL |
| `visible_context` | VARCHAR | 500 | 显性特征文本 |
| `visible_context_vector` | FLOAT_VECTOR | 1024 | 显性特征向量 |
| `hidden_context` | VARCHAR | 500 | 隐性特征文本 |
| `hidden_context_vector` | FLOAT_VECTOR | 1024 | 隐性特征向量 |
| `description` | VARCHAR | 2000 | 图片描述文本 |
| `description_vector` | FLOAT_VECTOR | 1024 | 图片描述向量 |

**设计特点：** 同时存储原始文本和向量，文本可作为可读的辅助信息，向量用于相似度计算。

##### `create_collection()` —— 创建集合并建立索引

为 `uid`、`image_id` 创建标量索引，为三个向量字段创建 **COSINE（余弦相似度）** 向量索引，使用 `AUTOINDEX` 自动选择最优索引类型。

##### `check_exists(uid, image_id)` —— 去重检查

通过标量过滤 `uid == xxx && image_id == xxx` 查询某用户是否已存过该图片，避免重复入库。

##### `insert_data(data)` —— 批量插入数据

将特征字典列表写入 Milvus 集合。

##### `search(uid, docs, topk)` —— 混合检索（核心）

这是本模块最关键的方法，实现了**三路混合检索 + 加权重排序**：

1. 对 `visible_context_vector`、`hidden_context_vector`、`description_vector` 三个向量字段分别发起 ANN（近似最近邻）搜索，每路召回 `topk * 3` 个候选
2. 使用 `WeightedRanker(0.4, 0.2, 0.4)` 对三路结果进行加权融合排序：
   - 显性特征权重：**0.4**
   - 隐性特征权重：**0.2**
   - 整体描述权重：**0.4**
3. 通过 `uid` 过滤确保只返回该用户的数据
4. 最终截取前 `topk` 条结果返回

**设计亮点：** 先多路召回扩大候选池，再加权重排序——兼顾了召回广度和排序精度。

### 4. `main.py` —— API 应用层

**职责：** FastAPI 应用入口，定义 RESTful API 端点，串联 feature_service 和 milvus_db 完成业务流程。

#### 启动初始化

- 创建 FastAPI 实例（title="Image RAG Demo"）
- 初始化 `MilvusModel` 单例（`auto_create=True`, `load_collection=True`），全局复用

#### API 端点

##### `POST /api/v1/images/rag` —— 图片入库

**流程：**

1. 接收 `SaveImageRequest`（uid + image_url）
2. 调用 `milvus_db.check_exists()` 去重，已存在的图片直接返回
3. 调用 `extract_features_from_image()` 提取三路语义特征 + 向量
4. 组装数据字典，调用 `milvus_db.insert_data()` 写入 Milvus
5. 返回成功消息

##### `POST /api/v1/recommend/text` —— 文本搜图

**流程：**

1. 接收 `TextRecommendRequest`（uid + query + top_k）
2. 调用 `extract_features_from_text()` 将搜索词扩写为三路查询向量
3. 调用 `milvus_db.search()` 执行混合检索，加权融合三路相似度
4. 提取结果中的 `image_id`（即图片 URL）列表
5. 返回 `RecommendResponse`（图片 URL 列表）

##### `POST /api/v1/recommend/image` —— 以图搜图

**流程：**

1. 接收 `ImageRecommendRequest`（uid + image_url + top_k）
2. 调用 `extract_features_from_image()` 提取参考图片的三路向量
3. 用这三路向量在 Milvus 中搜索相似图片
4. **过滤掉参考图片自身**，避免返回查询图本身
5. 返回 `RecommendResponse`

#### 启动方式

```bash
python main.py  # 启动在 http://0.0.0.0:8000
```

## 核心设计思想

### 1. 多路语义召回

传统的以图搜图仅基于单条向量做相似度匹配。本项目将每张图片从**三个语义维度**（显性/隐性/整体描述）进行拆解，搜索时三路并行召回再融合，大幅提升召回精度。

### 2. 文本查询扩写

用户输入的简短搜索词（如"温暖的咖啡馆"）通过 LLM 扩展成丰富的多维度描述，弥合了"用户简短查询"与"图片丰富语义"之间的语义鸿沟。

### 3. 用户级数据隔离

所有数据插入和检索都绑定 `uid` 过滤条件，不同用户的图片库完全隔离，支持多租户场景。

### 4. 加权混合检索

`WeightedRanker(0.4, 0.2, 0.4)` 赋予显性特征和整体描述更高权重（各 0.4），隐性特征权重较低（0.2）——体现了"看得见的东西"和"整体画面"比"主观感受"更可靠的业务判断。

### 1 分钟版本（项目概述）

**背景/要解决的问题：**
传统图像推荐系统（如 CLIP 端到端方案）输出单一向量，特征是黑盒的——你不知道模型到底关注了图片的什么。当业务需要"优先推荐同风格但不同内容的图片"或"只看光影构图相似的"时，单向量方案完全不可控。

**核心思路：**
利用 Qwen-VL-Max 的视觉理解与推理能力，通过结构化 Prompt 对每张图片强制提取 4 个维度的自然语言描述——**主体细节、场景环境、艺术风格、光影构图**。然后用 Qwen-Text-Embedding-v4 将每个维度转成 2048 维稠密向量，在 Milvus 中构建 Multi-Vector 集合（HNSW 索引）。检索时根据用户意图动态分配各维度权重，执行加权 Multi-Vector Search，实现细粒度可控的推荐。

**技术栈：**
Python + Qwen-VL-Max（视觉理解）+ Qwen-Text-Embedding-v4（2048D 向量化）+ Milvus（HNSW 索引 + Multi-Vector 检索）+ LangChain（流程编排）

**自己的角色：**
独立完成从架构设计、语义维度定义、Prompt 工程到全链路代码实现。

### 3-5 分钟版本（详细叙述）

#### 开场：一句话说清做什么

> "我做了一个基于视觉大模型语义解耦的 RAG 图像推荐系统。核心思路是把图片从 4 个语义维度拆开——主体、场景、风格、光影——各自向量化，检索时根据用户意图动态加权融合。解决的是传统方案'特征黑盒'和'不可控'的问题。"

#### 第一段：项目背景和痛点

> "传统图像推荐有两个核心问题。第一是特征黑盒：CLIP 这类模型输出单一向量，你知道两张图相似，但不知道为什么相似——是内容像还是风格像？这对推荐系统来说是致命的，因为你没法按业务需求做细粒度控制。第二是语义偏差：单向量把不同维度的信息混在一起，比如用户想要同一风格的不同内容，单向量可能优先返回内容相似的，风格反而被稀释了。"

#### 第二段：技术方案

> "我的方案分两条链路——离线入库和在线检索。"

> "**离线入库（Offline Indexing）：** 每张图片入库时，先用 Qwen-VL-Max 做语义解耦。通过精心设计的结构化 Prompt，强制模型从 4 个维度输出分析结果：
> - **主体细节（Subject）：** 画面中的核心对象、人物、物体特征
> - **场景环境（Context）：** 空间布局、环境氛围、背景信息
> - **艺术风格（Style）：** 绘画风格、色彩倾向、纹理质感
> - **光影构图（Composition）：** 光线方向、明暗对比、构图方式
>
> 然后拿这四段自然语言描述，分别过 Qwen-Text-Embedding-v4 转成 2048 维向量——比常见的 1024 维有更强的语义区分度。最后在 Milvus 里建一个 Multi-Vector 集合，四个向量字段各自建 HNSW 索引，用 COSINE 度量。"

> "**在线检索（Online Retrieval）：** 用户查询时走同样的四维拆解，拿到四路查询向量。核心创新点是动态权重引擎——不是固定权重，而是根据用户的行为意图来分配。比如用户在浏览艺术画廊类内容，系统自动把 Style 维度权重调高；如果用户在搜产品图，Subject 权重占主导。加权公式本质是各维度余弦相似度的线性加权和，4 个权重实时可调。"

#### 第三段：项目架构和工程细节

> "工程层面，我用 LangChain 做流程编排，把下载图片、调用 VLM 分析、四路向量化、插入 Milvus 串联成一条链。服务层是 FastAPI，暴露出 3 个接口：图片入库、文本搜图、以图搜图。整体代码量不大但链路完整，从 Prompt 设计到底层 HNSW 索引选型都覆盖了。"

#### 第四段：可能的延伸讨论（留给面试官追问的钩子）

> "如果要继续深入，可以聊几个方向：第一，动态权重目前怎么定——是规则引擎还是学出来的，如果上强化学习怎么做；第二，2048 维向量相比 1024 维的实际收益是多少，有没有做过消融实验；第三，4 个维度是不是最优解，如果加一个'情感维度'会不会提升。您对哪个方向感兴趣我可以展开。"

### 常见面试追问及应对

#### Q1: 为什么是 4 个维度，不是 3 个或 5 个？

> 这 4 个维度对应图片的四个独立且互补的语义层面：主体（拍的是什么）、场景（在哪拍的/什么环境）、风格（怎么呈现的）、光影（光线和构图怎样）。它们基本穷尽了一张图片能被语言描述的所有角度，且彼此不存在明显重叠。如果合并成 3 个——比如把光影并入场景——就会损失一个关键的可控维度（很多专业摄影师搜图就是按光影找的）。5 个以上的边际收益递减，增加的计算开销和 Prompt 复杂度不值得。

#### Q2: 动态权重引擎具体怎么实现？

> 核心逻辑是一套可配置的相似度聚合算法。当前阶段可以根据业务场景预设几组权重配置（比如"风格优先模式" Style 0.5 / Subject 0.1 / Context 0.2 / Composition 0.2），前端或上游推荐策略根据用户上下文选择对应配置。更进一步的方案是收集用户反馈信号（点击、收藏、停留时长），用 Learning to Rank 的思路离线学习最优权重，或者用 contextual bandit 在线调参。

#### Q3: 为什么选 2048 维而不是 1024 维？

> Qwen-Text-Embedding-v4 原生输出就是 2048 维。更高的维度意味着更大的语义容量和更精细的区分度——尤其在 4 个维度各有侧重的情况下，2048 维能更好地保留每个维度的细节信息。代价是存储和计算量翻倍，但 Milvus 的 HNSW 索引对 2048 维的支持很成熟，实际检索延迟增加在可接受范围内。

#### Q4: HNSW 相比 IVF 或其他索引算法的优势？

> HNSW 是目前 ANN 领域综合表现最好的算法之一——构建时逐层构建可导航小世界图，查询时贪婪搜索。相比 IVF，HNSW 不需要训练步骤，增量插入友好，召回率在同等性能下通常更高。代价是内存占用更大，但对于百万级图片库完全不是问题。

#### Q5: 这个方案相比直接用 CLIP 做端到端跨模态检索，优势在哪？

> CLIP 是黑盒单向量，适合通用场景。我的方案核心优势是可解释性和细粒度可控性。四路语义文本是可读的——debug 的时候你能明确看到模型把图片理解成了什么样。而且业务可以根据不同场景调配权重（比如设计类应用调高 Style 权重，电商应用调高 Subject 权重），CLIP 做不到这种灵活控制。本质上这是一个从"让模型替你决定相似"到"让业务定义相似"的范式转变。

#### Q6: LangChain 在项目里具体承担什么角色？

> 主要做流程编排——把图片下载、VLM 调用、Prompt 管理、四路向量化、Milvus 写入串联成一条可追踪的 Chain。LangChain 的结构化输出能力也用于解析 Qwen-VL-Max 返回的 JSON，做容错和降级。如果用原生代码写也能实现，但 LangChain 让流程更清晰、更易于扩展（比如后续加入缓存、重试、回调日志）。

#### Q7: 最难的/最有挑战的部分是什么？

> 选一个讲：
> - **概念设计层面：** 语义维度的定义和 Prompt 工程。要让 Qwen-VL-Max 稳定输出 4 个独立且不重叠的维度，迭代了多版 Prompt，最终靠结构化约束 + few-shot 示例才稳定下来。
> - **工程层面：** 四路向量入库 + 四路 ANN 加权检索的端到端延迟控制，以及 Milvus Multi-Vector 集合的 Schema 设计——4 个 2048 维向量字段加 4 个 VARCHAR 字段，索引策略要平衡召回率和写入性能。
